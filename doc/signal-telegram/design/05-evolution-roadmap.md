# TeamTalk 架构演进路线图

> 基于 Signal / Telegram 对比分析的综合性演进计划

---

## 演进原则

1. **办公 IM 定位不变**：面向中小型组织（<1万用户），单体架构，简单部署
2. **渐进式演进**：每个阶段可独立交付，不阻塞其他功能
3. **实用优先**：优先解决影响用户体验的问题，安全加固逐步推进
4. **保持简单**：不过度设计，每项改进都有明确的用户价值

---

## Phase 1：基础加固（当前 → MVP 稳定）

> 目标：让 TeamTalk 成为日常可用的 IM 工具

### 1.1 本地数据库引入

**问题**：每次打开聊天页都从服务器拉取全量消息，体验差、性能差。

**方案**：引入 SQLDelight（KMP 兼容的数据库库）

```
数据模型：
- messages(channel_id, seq, sender_uid, payload, message_type, timestamp, ...)
- conversations(channel_id, channel_type, name, last_message, unread_count, ...)
- contacts(uid, name, avatar, remark, ...)
- drafts(channel_id, content, updated_at)
```

**工作范围**：
- [ ] 添加 SQLDelight 依赖（commonMain）
- [ ] 定义 Schema 和数据模型
- [ ] 实现 Repository 层的双源策略（本地优先，网络同步）
- [ ] 消息本地缓存：拉取后写入本地，优先从本地读取
- [ ] 会话列表本地缓存
- [ ] 联系人列表本地缓存

### 1.2 Desktop 系统通知

**问题**：收到消息无任何提醒。

**方案**：ComposeNativeTray（原生系统托盘）+ 平台原生通知 API（详见 [09-notification-architecture.md](./09-notification-architecture.md)）

**工作范围**：
- [ ] 系统托盘图标（在线/离线状态 + 未读 badge）
- [ ] 新消息弹窗通知（显示发送者和消息摘要）
- [ ] 点击通知跳转到对应聊天
- [ ] 通知开关设置

### 1.3 消息搜索

**问题**：无法搜索历史消息。

**方案**：Apache Lucene 嵌入式全文搜索（详见 [10-fulltext-search.md](./10-fulltext-search.md)）+ 搜索 API

**工作范围**：
- [ ] 服务端：集成 Lucene，写入消息时同步建索引
- [ ] 服务端：`GET /api/v1/messages/search?q=xxx&channel_id=xxx`
- [ ] 客户端：搜索 UI（聊天内搜索 + 全局搜索）
- [ ] 搜索结果高亮关键词

### 1.4 消息同步优化

**问题**：多设备消息状态不同步。

**方案**：增强现有 `afterSeq` 增量拉取

**工作范围**：
- [ ] 会话列表增量同步（基于 `version` 字段）
- [ ] 已读回执多设备同步
- [ ] 离线期间消息补拉（记录最后 seq，上线后补拉）

## Phase 2：功能完善（MVP 稳定 → 日常可用）

> 目标：覆盖 IM 核心功能，满足日常办公需求

### 2.1 多设备在线

**问题**：当前同一账户只能一个设备在线。

**方案**：服务端支持多连接，每设备独立 seq 消费

**工作范围**：
- [ ] 服务端：`ConnectionManager` 支持同一 UID 多连接
- [ ] 服务端：消息投递到该 UID 的所有在线设备
- [ ] 客户端：设备管理页面（查看在线设备、远程注销）

### 2.2 语音消息

**方案**：平台原生录音 API + 文件上传

**工作范围**：
- [ ] Android：MediaRecorder 录音
- [ ] Desktop：Java Sound API 录音
- [ ] 录音 UI（长按录音、松开发送）
- [ ] 语音消息播放器
- [ ] 波形显示

### 2.3 在线状态 & 输入状态

**方案**：TCP 推送状态变更

**工作范围**：
- [ ] 服务端：维护用户在线/离线状态
- [ ] 服务端：TCP 推送状态变更通知
- [ ] 客户端：在线状态显示（绿点）
- [ ] 客户端：输入状态（"对方正在输入..."）

### 2.4 文件传输优化

**方案**：服务端校验 + 大小限制 + 进度反馈

**工作范围**：
- [ ] 服务端：文件大小限制配置
- [ ] 服务端：MIME 类型校验
- [ ] 服务端：图片缩略图自动生成
- [ ] 客户端：上传进度条
- [ ] 客户端：下载管理（重复下载检测）

### 2.5 消息编辑

**工作范围**：
- [ ] 服务端：`PUT /api/v1/messages/{id}` 更新消息内容
- [ ] 客户端：消息长按菜单增加"编辑"
- [ ] 消息编辑标记（显示"已编辑"）

### 2.6 Android 推送通知

**方案**：Firebase Cloud Messaging

**工作范围**：
- [ ] 服务端：集成 FCM SDK，向 Google 推送服务发送通知
- [ ] Android：FCM 接收，唤醒应用拉取消息
- [ ] Android：通知渠道配置（消息通知、群组通知）

## Phase 3：体验优化（日常可用 → 精品应用）

> 目标：打磨体验，追求品质

### 3.1 消息搜索增强

- [ ] 聊天内搜索（上下文导航，跳转到搜索结果位置）
- [ ] 全局搜索（跨所有聊天搜索）
- [ ] 按日期/发送者筛选
- [ ] 搜索结果上下文预览

### 3.2 群组功能增强

- [ ] 群管理员角色（创建者、管理员、普通成员）
- [ ] 群公告（置顶消息）
- [ ] 群禁言（全员/个人）
- [ ] @提及（@all、@特定成员）
- [ ] 群消息引用计数

### 3.3 性能优化

- [ ] 消息列表分页加载（RocksDB 游标分页）
- [ ] 图片懒加载 + 三级缓存
- [ ] 大文件断点续传
- [ ] WebSocket 传输（备选 TCP）

### 3.4 机器人平台

- [ ] Bot 框架设计（Webhook/消息订阅模式）
- [ ] Bot API（消息收发、群组管理）
- [ ] Bot 管理后台（创建/配置/权限）
- [ ] 内置 Bot 示例（欢迎消息、关键词回复）

## 技术选型建议

| 需求 | 推荐方案 | 备选 |
|------|----------|------|
| 本地数据库 | SQLDelight | Room (Android only) |
| 全文搜索 | Apache Lucene（嵌入式，详见 [10-fulltext-search.md](./10-fulltext-search.md)） | PostgreSQL tsvector |
| 推送通知 | FCM (Android) + ComposeNativeTray + 原生通知 API (Desktop) | — |
| 文件存储 | MinIO (已有) | 本地文件系统 |
| 音视频通话 | WebRTC | — |
| 传输加密 | TLS（Netty SslHandler） | — |

---

## 不在路线图中的项目

| 项目 | 原因 |
|------|------|
| 微服务架构 | 目标规模不需要，单体部署更简单 |
| CDN 加速 | 中小规模不需要 |
| 多数据中心 | 超出目标规模 |
| 频道/广播 | 与群组功能重叠，优先完善群组 |
| 贴纸系统 | 非核心功能 |
| E2EE（端到端加密） | 办公场景数据归组织所有，需满足合规审查 |
| 本地数据库加密 | 依赖 OS 底层安全（FileVault / BitLocker），应用层不重复加密 |
| 后量子加密 | 过度设计 |

## 与 CLAUDE.md 中目标的一致性

本文档的演进计划始终围绕 CLAUDE.md 中的定位：

> TeamTalk 的最终目标是实现一个对标钉钉、飞书的办公软件，面向中小型组织（用户规模一般不超过 1 万）。采用单体架构，几乎所有功能都可以用单机+内存的模型收敛到一个简单服务器上。

每个 Phase 都遵循这个定位：
- 不引入分布式组件（Kafka、Redis Cluster 等）
- 不使用云托管服务（DynamoDB、CDN 等）
- 保持 Kotlin 单一语言栈
- 单体部署，Docker Compose 一键启动
