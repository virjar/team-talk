# 安全架构对比分析

> Signal vs Telegram vs TeamTalk 安全机制

---

## 0. TeamTalk 安全定位

TeamTalk 是面向中小型组织的办公 IM，安全模型与个人隐私通讯工具有本质区别：

| 维度 | 个人隐私 IM（Signal） | 办公 IM（TeamTalk） |
|------|----------------------|-------------------|
| 数据归属 | 个人隐私，用户所有 | 组织资产，公司/机构所有 |
| 法律合规 | 个人隐私保护 | 符合国家法律法规，满足审查要求 |
| 信任模型 | 不信任服务端 | 信任组织自建的服务端 |
| 安全重点 | 防止服务端窃听 | 防止外部攻击者窃听（TLS 足够） |
| 本地安全 | 需要独立于 OS 的加密 | 依赖 OS 底层安全能力 |

**TeamTalk 安全策略**：TLS 传输加密 + OS 底层安全 + 基本的应用层认证鉴权，不做 E2EE 和本地独立加密。

---

## 1. 安全层级对比

| 安全层 | Signal | Telegram | TeamTalk |
|--------|--------|----------|----------|
| **传输加密** | TLS 1.3 | TLS + 自定义 AES-IGE | **无 TCP 加密（待引入 TLS）** |
| **消息加密** | 默认 E2EE | 仅 Secret Chat E2EE | 不需要（办公场景，组织信任服务端） |
| **密钥管理** | X3DH + Double Ratchet | DH + RSA | 不需要 |
| **身份验证** | Phone + PIN + Safety Numbers | Phone + Cloud Password | Phone + 验证码 + JWT |
| **本地加密** | SQLCipher | 无（Android） | 不需要（依赖 OS 安全） |
| **后量子安全** | CRYSTALS-Kyber | 无 | 不需要 |
| **元数据保护** | Sealed Sender | 无 | 不需要 |

## 2. TeamTalk 当前安全状态

### 2.1 已有的安全机制

| 机制 | 实现方式 | 安全等级 |
|------|---------|---------|
| 用户认证 | 手机号 + 短信验证码 + JWT | 基础 |
| API 鉴权 | Bearer Token（JWT） | 基础 |
| TCP 鉴权 | CONNECT 帧携带 token | 基础 |
| 密码哈希 | 服务端存储（bcrypt/argon2？需确认） | 未知 |
| HTTP API | 明文（开发模式） | **不安全** |

### 2.2 明确的安全缺陷

| 缺陷 | 风险等级 | 说明 |
|------|---------|------|
| **TCP 明文传输** | **高** | 所有消息内容在网络上明文传输，可被嗅探 |
| 固定验证码 | 高 | 开发模式 `TS_SMSCODE=123456`，生产环境需移除 |
| 无速率限制 | 中 | API 无请求频率限制，可被暴力攻击 |

> 注意：对于办公 IM 场景，"服务端可查看消息内容"和"设备丢失后本地数据可被读取"不属于安全缺陷——前者是组织资产的正常管理需求，后者依赖 OS 底层安全能力（设备锁、全盘加密等）。

## 3. 安全演进建议

### 3.1 短期（生产部署必须）

| 项目 | 优先级 | 方案 |
|------|--------|------|
| **TCP 传输加密** | **P0** | 直接使用 TLS（见下方分析） |
| 生产环境验证码 | **P0** | 接入真实短信服务，移除固定验证码 |
| API 速率限制 | **P0** | Ktor 插件实现 IP + UID 维度的速率限制 |
| 密码安全存储 | **P0** | 确认使用 bcrypt 或 argon2 |

### 3.2 TLS 传输加密

| 方案 | 复杂度 | 安全性 | 推荐度 |
|------|--------|--------|--------|
| **直接 TLS** | 低 | 高 | ★★★★★ |
| 应用层 AES-GCM | 中 | 高 | ★★★ |
| Signal 协议 | 高 | 极高 | ★ (过度) |
| MTProto 方案 | 高 | 中 | ★ (不推荐) |

**推荐：直接使用 TLS**（Netty 原生支持 SslHandler）
- 优点：标准化、工具链成熟、证书管理简单（内网可用自签名）
- 中小型组织内网部署：自签名 CA + 服务端证书
- 公网部署：Let's Encrypt 或商业证书

对于办公 IM 而言，TLS 提供的传输加密已经完全满足安全需求——防止网络中间人窃听和篡改。

### 3.3 中期安全增强

| 项目 | 优先级 | 说明 |
|------|--------|------|
| 设备管理 | P1 | 查看在线设备、远程注销 |
| JWT 刷新机制 | P1 | Token 过期 + Refresh Token 轮换 |

## 4. 不采用的安全设计

| 设计 | 原因 |
|------|------|
| E2EE（端到端加密） | 办公场景数据归组织所有，需满足合规审查，E2EE 反而阻碍管理和合规 |
| 本地数据库加密（SQLCipher） | 依赖 OS 底层安全（FileVault / BitLocker / Android 全盘加密），不需要应用层重复加密 |
| MTProto 的 AES-IGE | 非标准 AEAD，不如直接用 TLS |
| Telegram 的自定义加密层 | TLS 已经解决了传输加密问题 |
| Signal 的密封发送者 | 办公场景不需要对服务器隐藏发送者 |
| 后量子加密 | 过度设计 |
| 2FA 两步验证 | 办公场景由组织管理账号安全策略，不属于 IM 核心功能 |

### 关于 E2EE 不适用的说明

E2EE 的核心价值是"服务端无法解密消息内容"，这适用于个人隐私通讯（如 Signal）。但 TeamTalk 作为办公 IM：

1. **数据归属**：聊天记录和文件是组织资产，组织有权查看和管理
2. **合规要求**：聊天内容需符合国家法律法规，组织有义务保留和审查通讯记录
3. **管理需求**：管理员需要处理违规内容、审计通讯记录、响应法律取证
4. **技术复杂度**：E2EE 的密钥管理、多设备同步、群组加密等复杂度在办公场景下没有收益
