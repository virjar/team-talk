## 3. 关键缺失功能详解

### 3.1 本地数据库（最高优先级）

**现状**：每次打开聊天页都从服务器拉取全部消息列表。

**Signal/Telegram 的做法**：
- 所有消息本地 SQLite 缓存
- 打开聊天页从本地数据库读取，网络同步在后台进行
- 支持完全离线查看历史消息

**建议**：
- 引入 SQLDelight（KMP 兼容的数据库库）
- 核心表：messages、conversations、contacts、channels
- 同步策略：增量拉取（基于 seq 号）

### 3.2 消息搜索（高优先级）

**现状**：不支持任何搜索。

**Signal 做法**：
- 本地 SQLite FTS5 全文搜索
- 搜索消息内容、联系人、群组

**Telegram 做法**：
- 服务端搜索（支持海量消息）
- 本地缓存搜索

**建议**：
- 采用 Apache Lucene 嵌入式全文搜索（详见 [10-fulltext-search.md](../design/10-fulltext-search.md)）

### 3.3 多设备支持（高优先级）

**现状**：单设备登录，新设备登录会踢掉旧设备。

**Signal 做法**：
- 主设备 + 多个关联设备
- 主设备通过加密通道同步消息给关联设备
- 最多 5 个设备

**Telegram 做法**：
- 所有设备平等
- 云端存储消息，各设备独立同步
- 无设备数量限制

**建议**（参考 Telegram 模式更适合 TeamTalk）：
- 服务端存储所有消息
- 每个设备独立同步（基于 seq 号）
- 允许多设备同时在线

### 3.4 Desktop 系统通知（高优先级）

**现状**：收到消息无任何系统级通知。

**Signal Desktop**：Electron 原生通知 API
**Telegram Desktop**：Qt 系统通知

**建议**：
- Desktop：使用 ComposeNativeTray（原生系统托盘）+ 平台原生通知 API（详见 [09-notification-architecture.md](../design/09-notification-architecture.md)）
- 显示消息摘要、发送者名称
- 点击通知跳转到对应聊天

## 4. 功能演进优先级排序

### P0 — 必须有（用户体验基础）

1. **本地数据库** — 消息缓存、离线查看、性能提升
2. **Desktop 系统通知** — IM 应用基本要求
3. **消息搜索** — 用户找回历史消息的主要手段
4. **多设备支持** — 办公场景刚需（手机+电脑同时在线）

### P1 — 应该有（体验优化）

5. **语音消息** — IM 常见消息类型
6. **消息编辑** — 修正已发送消息
7. **在线状态** — 看到同事是否在线
8. **输入状态（typing）** — 聊天体验优化
9. **Android 推送通知** — 离线接收消息

### P2 — 可以有（锦上添花）

10. **群公告** — 组织信息发布
11. **群管理员角色** — 权限管理
12. **群禁言** — 群管理工具
13. **合并转发** — 消息整理分享
14. **机器人/Bot** — 办公自动化集成（详见 [01a-content-messages.md](../design/01a-content-messages.md) §3.10）

### P3 — 暂不需要

15. 音视频通话 — 可后期集成 WebRTC
16. 贴纸/表情 — 非核心功能
17. 频道/广播 — 与群组功能重叠，优先完善群组

### 不在路线图中

- **端到端加密（E2EE）** — 办公场景数据归组织所有，需满足合规审查，E2EE 阻碍管理和合规（详见 [04-security.md](../design/04-security.md)）
- **阅后即焚** — 同上，与组织数据管理需求矛盾
- **本地数据库加密** — 依赖 OS 底层安全（FileVault / BitLocker），应用层不重复加密
